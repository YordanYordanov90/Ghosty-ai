Migration **`drizzle/migrations/0001_youthful_the_hood.sql`** generated. **`POST /api/ai/design`** (`runtime = "nodejs"`): Clerk **`auth()`** + Zod-validate body → **`hasProjectAccess(projectId)`** → **`tasks.trigger<typeof designAgentTask>("design-agent", …)`** → insert **`taskRuns`** row (`runId`, `projectId`, `ownerId = userId`) → returns `{ runId }` with **202**. **`POST /api/ai/design/token`** (`runtime = "nodejs"`): Clerk auth + Zod-validate `{ runId }` → look up `task_runs` by `(runId, ownerId = userId)` → re-check `hasProjectAccess(record.projectId)` → **`auth.createPublicToken({ scopes: { read: { runs: [runId] } }, expirationTime: "1h" })`** → returns `{ token }`. **`proxy.ts`:** `isPublicRoute` adds **`/api/ai/design(.*)`** so handlers enforce JSON 401 themselves (no Clerk redirect for fetch clients). `npm run build` passes.
- **23-design-agent-logic** — **`lib/design-agent-payload.ts`**: wire schema allows optional **`projectId`**; **`normalizeDesignAgentPayload()`** resolves Drizzle **`projectId`** from explicit field or from UUID **`roomId`**, and **`liveblocksRoomId`** (UUID storage key for **`mutateFlow`**) falls back to **`projectId`** when **`roomId`** is non-UUID (manual Trigger tests). Invalid combos fail fast with **`AbortTaskRunError`** in the task or **400** from **`POST /api/ai/design`**. **`trigger/design-agent.ts`**: **`generateText`** (**`gpt-4o-mini`**) + **`extractJsonObject`** + **`designAgentPlanSchema.safeParse`** (**`lib/design-agent-schema.ts`** — discriminated actions; applied via **`applyDesignAgentActions`**) with **two attempts** (second prompt includes first validation errors); avoids OpenAI **`response_format`** / structured-json rejects on Zod **`discriminatedUnion`** schemas that **`generateObject`** produced. Canvas updates via **`mutateFlow`** + **`getLiveblocks()`**; progress via Trigger **`metadata`** + Liveblocks feed **`ghosty-design-agent-status`**; ephemeral **`setPresence`** cleared in **`finally`**; **`DesignAgentActivity`** (**`useFeedMessages`**). Trigger env needs **`OPENAI_API_KEY`** + **`LIVEBLOCKS_SECRET_KEY`**. Verified completing runs in Trigger.dev dev with payload **`{ roomId: <project-uuid>, prompt }`** only.

Liveblocks feed **`ai-status-feed`** (**`lib/ai-status-feed-constants.ts`**): ensured from sidebar via **`useCreateFeed`**; **`ArchitectChatPanelRoom`** uses **`useFeedMessages`**, shows latest **validated** message strip + disables textarea / spinner send while **`phase === 'active'`**; mock architect send posts **`active` → `complete`** feed messages + **`useUpdateMyPresence({ thinking })`** for cursor badges. **`workspace-layout.tsx`** wraps canvas + **`AiWorkspaceSidebar`** with **`LiveblocksProvider`** + **`RoomProvider`** (**`roomId`** prop passed); **`WorkspaceCanvas`** is room-inner only (**`WorkspaceCanvasLoading`** / **`WorkspaceCanvasErrorFallback`** exported). **`PresenceCursors`**: spinner on name badge when **`presence.thinking === true`**. - **27-spec-generation-flow** — Backend-only flow for AI spec generation. **`lib/spec-agent-payload.ts`**: Zod schemas for wire payload (`roomId`, `chatHistory`, `nodes`, `edges`) + task payload (adds server-resolved `projectId`) + `normalizeSpecAgentPayload()` (mirrors design-agent shape; project UUID resolved from `roomId` when it's already a UUID, falls back to explicit `projectId` for Trigger dashboard tests). **`trigger/generate-spec.ts`**: `generateSpecTask` (`schemaTask`, id `generate-spec`) — runs `generateText` (gpt-4o-mini via `@ai-sdk/openai`) over the canvas + chat snapshot, returns Markdown spec as task output (`{ ok, runId, spec, projectId }`); pushes phase + message updates through Trigger **`metadata`** and posts to room-scoped Liveblocks **`ai-status-feed`** (`taskKind: "spec"`, `phase: active|complete|error`) so the existing sidebar can gate input while a spec is running. **`POST /api/ai/spec`** (`runtime = "nodejs"`): Clerk `auth()` → Zod validate body (no client-supplied `projectId`) → resolve `projectId` from `roomId` → `hasProjectAccess(projectId)` → `tasks.trigger<typeof generateSpecTask>("generate-spec", …)` → insert `taskRuns` row scoped to (`runId`, `projectId`, `ownerId = userId`) → returns `{ runId }` with **202**. Trigger + DB insert are split into separate try/catches so a transient DB failure still surfaces the `runId`. **`POST /api/ai/spec/token`** (`runtime = "nodejs"`): Clerk auth + Zod-validate `{ runId }` → look up `task_runs` by (`runId`, `ownerId = userId`) → re-check `hasProjectAccess(record.projectId)` → `auth.createPublicToken({ scopes: { read: { runs: [runId] } }, expirationTime: "1h" })` → returns `{ token }`. **`proxy.ts`**: `isPublicRoute` adds **`/api/ai/spec(.*)`** so handlers respond with JSON 401 instead of redirecting fetch clients. `npm run build` passes; both routes appear in route manifest. No frontend, no spec storage, no spec editor UI in this unit.
- **28-spec-persistence-download** — Persist generated specs (metadata in Drizzle + content in Vercel Blob). **`drizzle/schema.ts`** adds `project_specs` (`id` uuid PK, `project_id` FK → `projects.id` ON DELETE CASCADE, `file_path` blob URL, `created_at`), indexes on `project_id` and (`project_id`, `created_at`); migration **`drizzle/migrations/0002_project_specs.sql`**. **`trigger/generate-spec.ts`** uploads generated Markdown to private Vercel Blob (`specs/{projectId}/{specId}.md`) and inserts `project_specs` row; task output includes `specId`. **Download route**: `GET /api/projects/[projectId]/specs/[specId]/download` authenticates with Clerk `auth()`, enforces `hasProjectAccess(projectId)`, verifies spec belongs to project, fetches private blob via SDK `get()`, returns Markdown as `Content-Disposition: attachment`. No frontend changes.
- **29-spec-ui-integration** — Specs tab in **`components/editor/ai-workspace-sidebar.tsx`** now loads spec list for current project workspace (`GET /api/projects/[projectId]/specs`) and shows compact scrollable list with `createdAt` + `filename`. Selecting a spec opens a shadcn **`Dialog`** preview modal that fetches Markdown via `GET /api/projects/[projectId]/specs/[specId]` (no direct Blob access client-side) and renders it with `react-markdown` + `remark-gfm` inside a shadcn **`ScrollArea`**. Download action available per list item and in modal via existing `GET /api/projects/[projectId]/specs/[specId]/download`.
- **30-spec-generate-button** — Specs tab **Generate Spec** now triggers **`POST /api/ai/spec`** with the current project `roomId`, room chat history, and live canvas nodes/edges snapshot, exchanges the returned `runId` for a scoped token via **`POST /api/ai/spec/token`**, subscribes with **`useRealtimeRun<typeof generateSpecTask>`**, shows a generating state/error message, and refreshes the specs list when the run completes. **`WorkspaceCanvas`** now reports its live nodes/edges snapshot up through **`WorkspaceLayout`** to **`AiWorkspaceSidebar`**. `npm run build` passes.

## Architecture Decisions

- shadcn/ui via CLI; do not hand-edit `components/ui/*` after generation (per feature spec).
- Next.js 16 + Clerk: use root **`proxy.ts`** (not `middleware.ts`) per Clerk quickstart and product spec.
- Auth UI: Clerk **`dark`** theme from `@clerk/ui/themes` (per `03-auth.md`), not Clerk's `shadcn` theme, with variables bridged to app tokens.
- **Home (`/`)** is a **public** route and hosts the **embedded `<SignUp />`** (path routing on `/`); **`/editor`** remains the post-auth workspace; dedicated **`/sign-in`** and **`/sign-up`** routes remain for deep links and parity with Clerk env paths.
- Editor project UI (**07**): orchestration in **`use-editor-project-actions`**; initial lists from **`fetchEditorProjectsData`** on the server; mutations via **`fetch`** to **`/api/projects`** with **`router.refresh()`**.
- Database (**05**): Drizzle schema lives under **`drizzle/schema.ts`**; app entrypoint **`lib/db.ts`** (Node `pg` pool; not Edge — use Neon serverless driver later if routes must run on Edge).
- Project REST (**06**): Handlers under **`app/api/projects`** validate with **Zod** and enforce owner checks on **`PATCH`** / **`DELETE`**; **`/api/projects`** is **public in `proxy.ts`** only so anonymous requests reach handlers and receive **`401` JSON**, not a sign-in redirect.
- Canvas persistence (**21**): Drizzle `projects.canvas_json_path` stores the Vercel Blob URL (metadata only); Vercel Blob stores the actual canvas JSON at `canvas/{projectId}.json` (private store, deterministic path, `allowOverwrite: true`); Liveblocks remains the real-time source of truth; blob is loaded only when Liveblocks room is empty at mount.
- Design generation (**22**): Trigger.dev runs are the source of truth for run state; Drizzle `task_runs` exists only to bind a `runId` to a Clerk `ownerId` + `projectId` so the token route can verify ownership before issuing a run-scoped public token (we don't mirror status/output in Postgres). Run tokens are read-only, scoped to a single `runId`, and short-lived (1h). `/api/ai/design(.*)` is **public in `proxy.ts`** only so handlers can return JSON 401 instead of a sign-in redirect (same pattern as `/api/projects`).
- Design agent (**23**): Canvas mutations from background tasks use **`mutateFlow`** (same Liveblocks `flow` storage as **`useLiveblocksFlow`**). Room-scoped feed **`ghosty-design-agent-status`** carries human-readable progress for all connected clients; Trigger **`metadata`** mirrors phases for **`useRealtimeRun`**. Liveblocks **`setPresence`** (REST) exposes AI cursor/thinking to collaborators without a browser session; **`OPENAI_API_KEY`** + **`LIVEBLOCKS_SECRET_KEY`** must exist on the Trigger worker environment. **Payload**: **`projectId`** optional when **`roomId`** is the project UUID; non-UUID **`roomId`** (e.g. dashboard test labels) requires explicit **`projectId`**. **LLM**: structured provider schemas from **`generateObject`** were rejected for **`designAgentPlanSchema`** (discriminated unions); validation runs client-side on **`generateText`** JSON instead.
- AI presence UI (**24**): Sidebar uses Liveblocks feed **`ai-status-feed`** (separate from **`ghosty-design-agent-status`**) for shared “generation active” state + optional status copy; payloads validated with **`types/tasks`** before UI. **`RoomProvider`** wraps both canvas and AI sidebar on project workspace; hub AI panel has no **`roomId`** and skips feed hooks.
- Spec generation (**27**): Same Trigger.dev + `task_runs` ownership model as design (**22**), but the API route never accepts a client-supplied `projectId` — `projectId` is resolved from `roomId` (must be the project UUID) before access check + trigger. Task output is plain Markdown returned in the run output (`{ ok, spec }`); we deliberately do **not** persist the spec in Postgres or Blob in this unit. Status updates flow through Trigger `metadata` (for `useRealtimeRun`) and the existing `ai-status-feed` with `taskKind: "spec"` so the sidebar's run-gating UI works for spec runs without any new frontend code. `/api/ai/spec(.*)` is **public in `proxy.ts`** only so handlers return JSON 401 instead of redirecting (same pattern as `/api/projects` and `/api/ai/design`).
